最近做的研究用到了，做一下备忘

Cache Allocation Technology(CAT)提供对给定线程，应用程序，虚拟机进行LLC级cache使用量限制
CAT引入了一种称为Class of Service( CLOS )的中间构造，可以理解为资源控制标签，无论是线程还是app/VM/container ，一旦被分组到这个标签下，并且此时CLOS如果又具有关联的resource capacity bitmasks(CBMs)，那么一组都能使用这个被约束的缓存。如图所示：

通过containing capacity bitmasks (CBMs)的MSR block指定给应用程序它可以使用的cache量. 这个所谓的CBMs表示了在L3 cache上不同线程重叠或者隔离程度。

举个形象化的例子，比如下图中，CLOS[1] 比CLOS[3]用的缓存更少，那么它就可以认为是较低的优先级。

在bitmask 不重叠的情况下，应用程序不会在L3层相互竞争空间，它们使用单独的分区，可以随着资源需求的变化随时动态更新.如上图中的CLOS[0] 和 CLOS[1]。
接下来我们考虑在重叠的情况下，如CLOS[2] 和CLOS[3] ，通常可以实现比不限制情况下更高的吞吐量，并且可以保留相对的优先级(显然此时CLOS[3] 比CLOS[2]优先级要高)

官方推荐的用法主要在云环境的优化，在多租户虚拟机的情况下，通常会有很多异构类型的应用程序。它们共享资源（就比如LL3 cache），但是某些应用程序（如视频流或转码）会消耗过多的LL3 cache. 例如下图中 “noisy neighbor”在core 0就占用了过多的LL3 cache, 在LL3这一级采用的是先来先服务的方式，此时尽管App[1]很重要，而且核心优先级要高，但是运行速度低于预期，性能倒转.

首先是怎么把CAT跑起来

分配并监视L3chache分配情况

  1. 配置重启内核 系统会开启RDT支持 通过 
    dmesg | grep rdt
    可以看到对应选项已经开启
  2. 手动挂载resctrl (通过resctrl文件系统来掌控分配)
    mount -t resctrl resctrl /sys/fs/resctrl
    mount（检查是否挂载上） 如显示resctrl on /sys/fs/resctrl type resctrl (rw,relatime)则已经成功挂载

  3. resctrl 下的几个文件

    cpus: 16进制整数，每个bit代表一个核心 cat cpus
schemata: 策略表，每个策略为一行，格式为: L3:<cache_id0>=<cbm>
info/L3/cbm_mask: 当前CPU支持的L3cache单元数量
info/L3/min_cbm_bits：分配给某个策略组的最小cbm数量
info/L3/num_closids: 该系统最多可支持策略组的数量
tasks: 采用该策略的所有进程

  4. cd /sys/fs/resctrl

  5. mkdir p1 //创建策略P1
  6. echo f > cpus //分配的核心
  7. echo L3:0=3 > schemata //分配L3 （咱们的机器只有一个插槽 所以socket = 0就行）
  8. echo 1000 > tasks //将进程分配给策略P1
  9. cat /sys/fs/resctrl/p1/mon_data/mon_L3_00/llc_occupancy //llc占用快照 单位为字节
